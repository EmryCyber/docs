When an API request fails, Regentra returns a JSON error response with an HTTP status code, a machine-readable error code, and a human-readable message. Use the error code to handle failures programmatically and the message to understand what went wrong.

## Error response format

All error responses follow this structure:

```json
{
  "error": {
    "code": "resource_not_found",
    "message": "The requested resource could not be found.",
    "request_id": "req_abc123"
  }
}
```

<ResponseField name="error.code" type="string">
  Machine-readable error code. Use this in your code to identify and handle specific failure types.
</ResponseField>

<ResponseField name="error.message" type="string">
  Human-readable description of what went wrong.
</ResponseField>

<ResponseField name="error.request_id" type="string">
  Unique identifier for the request. Include this value when contacting Regentra support — it helps the team locate and trace your request quickly.
</ResponseField>

## HTTP status codes

| Status | Meaning | Common causes |
|---|---|---|
| `400` | Bad Request | Invalid request body or missing required fields |
| `401` | Unauthorized | API key missing or invalid |
| `403` | Forbidden | API key lacks the required scope for this action |
| `404` | Not Found | Resource doesn't exist or the ID is incorrect |
| `409` | Conflict | Request conflicts with existing data (for example, a duplicate name) |
| `422` | Unprocessable Entity | Validation error — check field constraints in the response details |
| `429` | Too Many Requests | Rate limit exceeded. Back off and retry after the `Retry-After` header value. |
| `500` | Internal Server Error | Unexpected server-side error. Retry the request; if the error persists, contact support with the `request_id`. |
| `503` | Service Unavailable | Temporary outage. Check the [Regentra status page](https://status.regentra.io) and retry. |

## Error codes reference

| Code | Description | How to fix |
|---|---|---|
| `invalid_api_key` | The API key format is incorrect or the key does not exist | Check that your key starts with `rg_live_` or `rg_test_` and that it was copied correctly |
| `insufficient_scope` | The API key does not have permission to perform this action | Create a new key or update your existing key with the required scope in **Settings → API Keys** |
| `resource_not_found` | No resource with the given ID exists in your account | Verify the ID is correct and that the resource has not been deleted |
| `validation_error` | One or more fields failed validation | Check the `details` array in the error response for field-level messages |
| `rate_limit_exceeded` | You have sent too many requests in a short period | Wait until the time specified in the `Retry-After` response header, then retry |
| `internal_server_error` | An unexpected error occurred on Regentra's servers | Retry the request. If it continues to fail, contact support with the `request_id` value |

<Note>
  For `validation_error` responses, the error object may include a `details` array with one entry per invalid field. Check each entry's `field` and `message` properties to identify which inputs need to be corrected.
</Note>

## Handling errors

The example below shows how to check for errors and handle specific cases such as rate limiting:

```javascript
const response = await fetch('https://api.regentra.io/v1/resources/res_123', {
  headers: { 'Authorization': 'Bearer YOUR_API_KEY' }
});

if (!response.ok) {
  const { error } = await response.json();
  console.error(`Error ${response.status}: ${error.code} — ${error.message}`);

  // Handle specific codes
  if (response.status === 429) {
    const retryAfter = response.headers.get('Retry-After');
    // Wait the specified number of seconds, then retry the request
  }

  if (response.status === 401) {
    // Prompt re-authentication or check API key configuration
  }
}
```
