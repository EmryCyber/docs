---
title: "Manage environments for staging and production"
description: "Use Regentra environments to keep staging and production configurations, API keys, and resources completely isolated from each other."
---

Regentra environments let you maintain separate configurations, API keys, and data for different stages of your development lifecycle — typically staging and production. This isolation means you can test changes safely without risking your live setup.

## What are environments?

Each environment in Regentra is a fully independent context. It has its own:

- **API keys** — credentials scoped exclusively to that environment
- **Resources** — data and assets that don't cross environment boundaries
- **Workflow configurations** — trigger settings, step parameters, and integration credentials

Changes you make in one environment never affect another. You can freely experiment in staging and only promote changes to production when you're confident they work correctly.

## Creating an environment

<Steps>
  <Step title="Open environment settings">
    Go to **Settings → Environments** in the dashboard.
  </Step>
  <Step title="Create a new environment">
    Click **New Environment**.
  </Step>
  <Step title="Name the environment">
    Enter a descriptive name, such as "Staging" or "Production". Names must be unique within your workspace.
  </Step>
  <Step title="Configure environment settings">
    Set any environment-specific options, such as rate limits, allowed integrations, or notification preferences.
  </Step>
  <Step title="Generate API keys">
    Click **Generate API Key** to create credentials for this environment. Store the key securely — it's only shown once.
  </Step>
</Steps>

## Switching environments

The active environment is shown in the top navigation bar. Click the environment name to open the switcher and select a different one. All dashboard views — resources, workflows, logs — update to reflect the selected environment.

## Environment-specific API keys

Every environment has its own set of API keys. Your application must use the correct key for each environment; using a staging key against production endpoints (or vice versa) will result in authentication errors.

## Best practices

<AccordionGroup>
  <Accordion title="Test in staging before promoting to production">
    Always validate workflow changes, new integrations, and configuration updates in staging first. Once you're confident the behavior is correct, apply the same changes to production.
  </Accordion>
  <Accordion title="Restrict production API key access">
    Apply the principle of least privilege: give production keys only the scopes they need. Avoid creating broad read-write keys unless required.
  </Accordion>
  <Accordion title="Never use production data in staging">
    Avoid copying real customer data into staging environments. Use synthetic or anonymized data for testing to reduce risk and comply with data protection obligations.
  </Accordion>
</AccordionGroup>

<Tip>
  Store API keys in environment variables in your application rather than hardcoding them. This makes it easy to switch between environments without changing code.
</Tip>

```bash
REGENTRA_API_KEY=rg_live_xxxxx   # production
# REGENTRA_API_KEY=rg_test_xxxxx  # staging
REGENTRA_API_URL=https://api.regentra.io/v1
```
